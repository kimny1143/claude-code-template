# Tier 3: コンダクターレビュー運用ガイド

## 対象となる変更

以下のいずれかに該当するPRは全てTier 3:

| 基準 | 具体例 |
|------|--------|
| **価格・原価率に影響** | Stripeの価格設定変更、原価率パラメータ変更、プラン構成変更 |
| **認証・セキュリティ** | OAuth設定、APIキー管理、トークン処理、CORS設定、RLS変更 |
| **DBスキーマ変更** | マイグレーション追加、テーブル追加・変更・削除、インデックス変更 |
| **本番環境設定** | Vercel環境変数、Cloudflare Workers設定、GitHub Actions secrets |
| **新しい外部サービス導入** | 新API連携、新SaaSの導入、新決済手段の追加 |
| **全課波及の変更** | template課からの変更（setup.sh、共通スキル、共通設定） |

## なぜTier 3が必要か

Tier 1/2で判断を委譲しても、上記の変更は**判断ミスのコストが非常に高い**:

- 価格設定ミス → 売上直撃
- セキュリティホール → データ漏洩リスク
- DBスキーマ破壊 → サービス停止
- template課の変更 → 全課の動作に影響

これらはコンダクター（管理インスタンス）+ CEO（人間）のダブルチェックで守る。

## Tier 3 PR フロー

```
課がPRを作成
  │
  ├─ claude-peersでコンダクターにPR URL通知
  │
  ├─ コンダクターがレビュー
  │   ├─ CI全通過チェック
  │   ├─ 変更内容の妥当性確認
  │   ├─ 影響範囲の特定
  │   └─ 関連課への影響通知（必要に応じて）
  │
  ├─ コンダクターが承認 or フィードバック
  │
  ├─ CEO（kimny）が最終承認
  │
  └─ マージ
```

### 1. PR作成と通知

```bash
# PR作成
gh pr create --title "feat: add Stripe webhook handler" \
  --body "## Summary
- Stripe Webhook エンドポイント追加
- subscription.created / subscription.updated / subscription.deleted を処理

## Tier 3 理由
- 新しい外部サービス連携（Stripe Webhook）
- 認証処理（Webhook署名検証）

## Test plan
- [ ] Stripe CLI でローカルテスト
- [ ] 署名検証が正しく動作すること
- [ ] 各イベントのハンドリングが正しいこと"
```

```
# claude-peersでコンダクターに通知
【mued課→conductor課 Tier 3 PR提出】
PR: https://github.com/xxx/xxx/pull/55
Tier 3理由: Stripe Webhook連携（外部サービス + 認証）
確認ポイント: Webhook署名検証の実装、エラーハンドリング
```

### 2. コンダクターのレビューチェックリスト

コンダクターは以下を確認:

#### セキュリティ関連
- [ ] 認証情報がハードコードされていないか
- [ ] `.env` に新しいシークレットが必要な場合、`.env.example` が更新されているか
- [ ] APIキーやトークンの露出リスクがないか
- [ ] 入力バリデーションが適切か

#### DB関連
- [ ] マイグレーションが可逆か（rollbackスクリプトの有無）
- [ ] 既存データへの影響はないか
- [ ] インデックスのパフォーマンス影響

#### 外部サービス関連
- [ ] コスト影響の見積もりがあるか
- [ ] 無料枠の範囲内か、超過時のコストは
- [ ] 代替サービスの検討がされたか
- [ ] 障害時のフォールバック

#### 全課波及関連（template課の変更）
- [ ] 後方互換性があるか
- [ ] 各課への影響範囲が明確か
- [ ] 段階的ロールアウトが可能か

### 3. CEO承認

コンダクターが承認した後、CEO（kimny）が最終確認:

- PR内容の確認（`gh pr view <番号>`）
- ビジネス上の妥当性確認
- マージ承認

### 4. マージ後

- コンダクターがpatrol報告に記録
- 影響を受ける課に通知（template課の変更の場合は全課）

## Tier 3 → Tier 2 への降格

以下の条件を**全て**満たす場合、コンダクター判断でTier 2に降格できる:

1. 変更が軽微（typo修正レベルの設定変更等）
2. 影響範囲が明確に限定的
3. 可逆である（すぐに戻せる）

降格した場合もpatrol報告に記録する。

## 緊急時の対応

### ホットフィックスが必要な場合

1. PR作成 + コンダクターに **[URGENT]** タグで通知
2. コンダクターが即座にレビュー
3. CEO不在の場合: コンダクター承認でマージ可（事後報告必須）

### ロールバックが必要な場合

1. 問題を発見した課がコンダクターに即報告
2. コンダクターがrevert PRを作成
3. revert PRはTier 3だが、緊急扱いで即マージ可
