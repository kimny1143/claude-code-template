# Tier 2: ピアレビュー運用ガイド

## 対象となる変更

- Tier 3に該当しないコード変更全般
- コンポーネントの追加・修正
- ユーティリティ関数の追加・修正
- スタイル変更（CSS/Tailwind）
- 設定ファイルの変更（`.gitignore`, `tsconfig.json` 等）
- 依存関係の更新（セキュリティパッチ等）
- リファクタリング

## ピアレビューの仕組み

### レビュー担当の決め方

同部署内の別課を相互レビュー相手に指定する。

**設計理由:**
- 同部署なのでコードベースへの理解がある
- 別課なので「自分が書いたコードを自分で承認する」問題を回避
- 相互関係なので負担が偏らない

**例（glasswerks AI Room の場合）:**

| 自分 | レビュー依頼先 | 理由 |
|------|-------------|------|
| mued課 | native課 | 同じプロダクト部。技術スタック（React）が共通 |
| native課 | mued課 | 同上 |
| SNS課 | write課 | 同じマーケティング部。コンテンツ配信の上下流 |
| write課 | SNS課 | 同上 |
| video課 | SNS課 | マーケティング部。動画の配信先がSNS |
| LP課 | mued課 | フロントエンド技術が共通 |
| data課 | mued課 | データ分析対象がmued |
| freee課 | LP課 | コーポレート部の相互 |

### あなたのチームに適用するには

1. 各課を部署ごとにグルーピング
2. 部署内でペアを組む（奇数なら3課ローテーション）
3. 部署が1課しかない場合は、最も技術スタックが近い他部署とペア

## ピアレビュー手順

### レビュー依頼側（PR作成者）

#### 1. PRを作成

```bash
gh pr create --title "[peer-review: native課] feat: add search component" \
  --body "## Summary
- 検索コンポーネントを追加
- デバウンス処理付き

## Test plan
- [ ] 検索入力でAPIが呼ばれること
- [ ] デバウンスが効いていること"
```

#### 2. ピア課にレビュー依頼を送信

claude-peersで直接連絡:

```
【mued課→native課 レビュー依頼】
PR: https://github.com/xxx/xxx/pull/42
変更概要: 検索コンポーネント追加（Tier 2）
確認ポイント: デバウンス処理のタイミングが適切か
```

#### 3. フィードバック対応

- ピアからのフィードバックがあれば対応
- 対応完了後、再度ピアに通知

### レビュー側（ピア課）

#### 1. レビュー通知を受け取る

claude-peersメッセージで依頼を受領。

#### 2. レビュー実施

**チェック項目:**

- [ ] 変更がTier 2の範囲内か（Tier 3に該当しないか）
- [ ] CIが全通過しているか
- [ ] コードが既存パターンに沿っているか
- [ ] 明らかなバグやセキュリティ問題がないか
- [ ] テストが追加・更新されているか

**レビューの深さ:**
- 全行を精読する必要はない
- 「別の課が見ても明らかな問題がないか」のチェック
- アーキテクチャ議論は必要ならTier 3に昇格させる

#### 3. 承認またはフィードバック

```
【native課→mued課 レビュー完了】
PR #42: ✅ 承認。マージOKです。
```

または:

```
【native課→mued課 レビューフィードバック】
PR #42: デバウンスのタイミングが300msだが、検索APIのレスポンスタイムを考慮すると500msの方が安全では？確認お願いします。
```

#### 4. マージ

承認後、PR作成者がマージ:

```bash
gh pr merge 42 --squash
```

## 所要時間の目安

| PR規模 | レビュー時間 | 目安 |
|--------|-----------|------|
| 小（~50行） | 1-2分 | 関数追加、スタイル修正 |
| 中（50-200行） | 3-5分 | コンポーネント追加 |
| 大（200行+） | 5-10分 | 機能追加、リファクタリング |

**大きすぎるPRはTier 3昇格を検討。** 200行超のPRは分割も検討する。

## エスカレーション

以下の場合、ピアレビュー中でもTier 3に昇格させる:

- レビュー中にセキュリティリスクを発見
- 変更の影響範囲がTier 2の想定を超える
- ピア間で判断が割れた場合
- レビューに30分以上かかりそうな複雑さ
