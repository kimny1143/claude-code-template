# カスタマイズガイド

## 課の数を決める

### 推奨パターン

| チーム規模 | 課数 | 構成例 |
|-----------|------|--------|
| **ミニマル（個人開発）** | 3課 | conductor + app + content |
| **スタンダード（小規模チーム）** | 5-6課 | conductor + frontend + backend + content + infra |
| **フル（組織運営）** | 8-12課 | glasswerksパターン（10課 + 拡張） |

### 課を増やすタイミング

- 1つの課のPR数が週10+を超える → 分割を検討
- 2つの異なる技術スタックを1課が担当 → 分割
- レビュー待ちがボトルネック → 並行処理のため分割

### 課を減らすタイミング

- 週にPRが1-2件しかない課 → 統合を検討
- 技術スタックが同じ2課 → 統合
- コミュニケーションコストが生産性を上回る → 統合

## conductor課（管理インスタンス）の設計

conductor課は必須。以下の役割を担う:

1. **巡回（patrol）**: 定期的に全課のステータスを確認
2. **タスク振り分け（dispatch）**: 新タスクを適切な課に割り当て
3. **PRレビュー**: Tier 3 PRのレビュー
4. **横連携**: 課間の依存関係・ブロッカーの解消
5. **レポート**: 日次・週次の進捗サマリー

### conductor課のCLAUDE.mdに必要な要素

- 全課のワークスペース一覧
- 巡回のスケジュールと手順
- PRレビューのチェックリスト
- エスカレーション基準
