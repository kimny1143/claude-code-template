# 設計判断メモ: AIエージェント10課体制の構成理由

このドキュメントでは、glasswerks AI Roomがなぜこの構成になっているのか — 各課の設置理由、検討したトレードオフ、そして自分のチームに適用する際のガイダンスを説明する。

---

## 1. なぜ部門に分けるのか

### 単一エージェントの限界

1つのClaude Codeインスタンスですべて — コード、コンテンツ、デプロイ、マーケティング — を処理しようとすると、すぐに実用上の限界にぶつかる:

- **コンテキストウィンドウの汚染。** 200Kトークンの会話は、同一セッション内でNext.jsコンポーネント、Threads API呼び出し、記事ドラフトを同時に扱うとすぐに埋まる。エージェントは以前の判断を見失い始める。
- **指示の矛盾。** 「簡潔で技術的なコードコメントを書け」と「温かみのある会話調のマーケティングコピーを書け」は矛盾するスタイルガイドだ。1つのCLAUDE.mdで両方をうまく処理することはできない。
- **並列処理ができない。** エージェントが1つだと作業は逐次実行になる。ブログ記事を書いている間、バグ修正は待たされる。

### 課モデル

各課（department）は独立したClaude Codeインスタンスであり、以下を持つ:
- 専用のCLAUDE.md（課に最適化された指示、ツール、トーン）
- 専用の作業ディレクトリ（またはモノレポ内の特定スコープ）
- claude-peersによる通信（状態の共有ではなくメッセージパッシング）

これにより **専門化**（各エージェントが1つのことに秀でる）、**並列処理**（複数エージェントが同時に作業）、**分離**（1つの課のミスが他の課のコンテキストを破壊しない）が実現する。

## 2. 10課の構成とそれぞれの存在理由

### プロダクト開発（3課）

| 課 | 責任範囲 | なぜ独立しているか |
|----|---------|-------------------|
| **mued課** | Webアプリケーション（Next.js、Prisma、Supabase） | フロントエンド＋バックエンド＋DBマイグレーションだけでフルの作業量。データモデルとUIパターンへの深いコンテキストが必要。 |
| **native課** | ネイティブ/モバイルアプリケーション | ツールチェーンが異なる（Swift/Kotlin vs TypeScript）。デプロイパイプラインもレビュー頻度も異なる。Webとコンテキストを共有するのは無駄。 |
| **template課** | 再利用可能なテンプレート、ボイラープレート、共有設定 | ここでの変更は全課に影響する。独自のレビュープロセス（常にTier 3）と慎重なバージョニングが必要。 |

**mued課とnative課を統合しない理由:** プロダクトは共有しているが、技術スタックは共有していない。1つのエージェントがNext.jsとSwiftを行き来すると効率が落ちる。別々のエージェントなら同一機能で並列作業が可能（例: mued課がAPIを構築し、native課がモバイルクライアントを構築）。

### コンテンツ＆マーケティング（4課）

| 課 | 責任範囲 | なぜ独立しているか |
|----|---------|-------------------|
| **write課** | 長文コンテンツ: 記事、ドキュメント、App Storeコピー | 執筆品質の維持には、トーンガイド、スタイルルール、編集ワークフローを含む専用CLAUDE.mdが必要。コード生成の指示と混ぜると両方の品質が下がる。 |
| **SNS課** | SNS運用: スケジューリング、投稿、アナリティクス連携 | 運用フォーカス — JSONデータファイル、cronスケジュール、API連携の管理。コンテンツ制作とはリズムが異なる。 |
| **video課** | 動画スクリプト、サムネイル、ビジュアルアセット | 画像/動画生成はトークン消費が大きい。分離することで、コストスパイクが他の課に影響するのを防ぐ。 |
| **LP課** | ランディングページとコンバージョン最適化 | コード（ページ構築）とマーケティング（コピー/レイアウト最適化）の中間に位置する。CROフレームワークとフロントエンドスキルの両方がコンテキストに必要。 |

**write課とSNS課を分ける理由:** 記事を書くことと、Threadsの投稿をスケジュールすることは異なるワークフロー。write課は品質と編集プロセスに集中し、SNS課は配信インフラとタイミングに集中する。受け渡しポイント（write課がドラフト → SNS課が公開）は明確で監査可能。

### オペレーション＆データ（3課）

| 課 | 責任範囲 | なぜ独立しているか |
|----|---------|-------------------|
| **conductor課** | オーケストレーション: PRレビュー、タスク配分、部門間ブロッカー解消、デイリーダイジェスト | conductorはプロダクションコードを書かない。仕事は調整。調整と実装を混ぜると利益相反が生じる（自分のコードを自分でレビュー）。 |
| **data課** | アナリティクス、メトリクス、ダッシュボード、データパイプライン | 専門ツール（Snowflake、データ可視化）。独自のスキーマ知識とクエリパターンが必要。 |
| **freee課** | 会計連携（freee API） | 財務データには厳格なアクセス制御が必要。分離することでミスの影響範囲を限定し、財務APIの認証情報を他課のコンテキストから排除する。 |

**conductor課はなぜワークフローではなく課なのか?** 調整は一度きりのステップではなく、継続的な作業だから。conductorは全課のブロッカーを追跡し、デイリーダイジェストを作成し、Tierレビュー制度を管理し、コスト懸念をエスカレーションする。課にすることで、永続的なコンテキストと明確な通信チャネルが得られる。

## 3. 責任範囲の決め方

### 境界テスト

各作業について以下を問うた: **「このエージェントのコンテキストウィンドウが完全にリセットされた場合、CLAUDE.mdとリポジトリの状態だけでこの作業を再開できるか?」**

Yesなら → その課に属する。
Noなら → CLAUDE.mdにもっとコンテキストが必要か、作業が別の課に属している。

### 受け渡しテスト

2つの課が連携する際: **「課間で受け渡される明確な成果物はあるか?」**

- write課 → SNS課: テキストドラフト（claude-peers経由のMarkdown文字列）
- mued課 → data課: アナリティクスイベントスキーマ（コミットされた仕様ファイル）
- LP課 → write課: コピーブリーフ（制約付きの構造化リクエスト）

「成果物」が曖昧な口頭指示（「もっと良くして」）になるなら、境界の位置が間違っている。

### コスト分離テスト

**「この課が通常予算の3倍を消費した場合、他の課に影響するか?」**

画像生成の多い作業（video課）やデータパイプライン作業（data課）を独立した課に分離することで、1課のコストスパイクが他を圧迫しない。conductorが総支出を監視し、異常を検知する。

## 4. 迷った判断

### LP課はmued課の一部にすべきか

**統合の論拠:** LPは技術的にはWebアプリケーションの一部。同じスタック、同じデプロイ。

**分離の論拠（採用した判断）:** LP作業はマーケティング指標（コンバージョン率、直帰率）で駆動されるのであり、プロダクト指標（機能完成度、バグ数）ではない。LP課のCLAUDE.mdにはCROフレームワーク、A/Bテストパターン、コピーライティング原則が含まれており、mued課の指示を煩雑にする。境界は明確: mued課がアプリケーションを所有し、LP課がマーケティングサイトを所有する。

### 専任のQA/テスト課を設けるべきか

**設置の論拠:** テスト戦略の一元化、カバレッジ基準の統一。

**不設置の論拠（採用した判断）:** 各課が自分のテストを書く。何をテストすべきか最も深いコンテキストを持っているのは各課自身だから。conductorは「CIが通らないとマージ不可」を共通ルールとして強制する。この規模では、専任QA課は受け渡しオーバーヘッドを生む割に、それに見合う価値を追加しない。

### conductor課にコードを書かせるべきか

**許可の論拠:** チームのブロックを解消する最速の方法が、5行の修正を自分で書くことである場合がある。

**禁止の論拠（採用した判断）:** 関心の分離。conductorはコードをレビューするが、書かない。conductorが書くこととレビューすることの両方を行えば、独立した検証が失われる。コストは時折の遅延。ベネフィットはレビュープロセスへの信頼。

### 10課は多すぎるか

一見すると10は重い。しかし:
- 各課はClaude Codeインスタンスであり、人間の雇用ではない。課を1つ追加する限界費用は月額約$0。
- 本当のコストは **調整オーバーヘッド**（claude-peersメッセージ、PRレビュー、ブロッカー解消）。conductorはまさにこれを管理するために存在する。
- **過少分割**（例: write + SNS + LP + videoを1つの「マーケティング」課で処理）はコンテキストウィンドウの枯渇と指示の矛盾を招いた。10課への過剰分割のほうが、コンテキスト汚染によるデバッグ時間の損失より安上がりだった。

## 5. 自分のチームへの適用ガイド

すべてのチームが10課必要なわけではない。適正規模の決め方を以下に示す。

### 3課から始める（最小実行可能な組織）

| 課 | カバー範囲 |
|----|-----------|
| **conductor** | オーケストレーション、レビュー、デイリーダイジェスト |
| **dev** | 全プロダクト開発 |
| **content** | 全コンテンツ、マーケティング、コミュニケーション |

個人開発者や、コードベースが1つでマーケティングが軽いプロダクトに適する。

### 痛みを感じたら5課にスケール

以下に気づいたら分割する:
- **dev**のコンテキストがあふれている → スタック別（フロントエンド/バックエンド）またはプロダクト領域別に分割
- **content**が多すぎるフォーマットをさばいている → write（長文）とops（スケジューリング、投稿）を分割
- 特定の連携（決済、アナリティクス、会計）に分離された認証情報が必要 → 専用課を設ける

### 複数プロダクトがあれば10課にスケール

10課体制は以下の場合に適する:
- 2つ以上のプロダクトまたはプラットフォーム（Web + モバイル）
- アクティブなコンテンツマーケティング（記事 + SNS + 動画）
- 分離が必要な財務/コンプライアンス連携
- 1つのconductorが調整だけで忙しくなるほどの日次作業量

### 最初に変えるべきもの

1. **課名。** 自分のドメインに合う名前を使う。「mued課」は我々のプロダクト固有の名前であり、あなたの場合は「api-team」や「dashboard-dept」かもしれない。
2. **Tierの閾値。** 我々のTier 3は「新規外部API」を含む。あなたの場合は「課金ロジックの変更」や「ユーザー向けメールテンプレートの変更」かもしれない。
3. **ピアレビューのペア。** Tier 2レビュー表はチームのトポロジーに合わせる。互いのコンテキストを理解している課同士をペアにする。
4. **conductorのスコープ。** 3課構成ではconductorがops作業も兼任できる。10課ではconductorは調整のみに専念すべき。

### 分割を間違えたサイン

- 2つの課が常に互いのファイルを必要とする → 統合する
- ある課がほとんどの日にアイドル状態 → 隣の課に統合する
- conductorがPRレビューよりメッセージのルーティングに時間を使っている → 課が多すぎるか、境界が不明確
- ある課のCLAUDE.mdが500行を超えている → カバーする関心が多すぎる可能性。分割を検討

## 6. 構造の背景にある原則

1. **専門化するのはCLAUDE.mdであり、モデルではない。** 全課が同じClaudeモデルを使う。差別化は指示、ツール、コンテキストにあり、能力にはない。

2. **会話ではなく成果物で通信する。** 課間ではファイル、PRレビュー、構造化メッセージをやり取りする。フリーフォームのチャットではない。これにより受け渡しが監査可能で再開可能になる。

3. **conductorは作らない。作り手は調整しない。** 実装とオーケストレーションの分離は利益相反を防ぎ、レビュー品質を高く保つ。

4. **便利さではなく影響範囲で分離する。** 財務連携や部門横断テンプレートが専用課を持つのは、作業が複雑だからではなく、ミスが高くつくからだ。

5. **課を追加するのはコンテキストがあふれたとき。人数が増えたときではない。** 分割のトリガーは「このエージェントが以前の判断を忘れ続ける」や「これらの指示が互いに矛盾する」であり、「作業が増えた」ではない。
