## キャラクター gate — 機械テストで拾えない人間判断の合格基準

「テスト通過」 の意味が 「コードが壊れていない」 (型 / lint / build / コードレビュー LGTM) に偏り、 「成果物が実際に使える」 (画面が見える・操作できる・遊べる・読める・聴ける) を含まない —— この曖昧さが誤認の温床になる。 MUEDear Build 78 事故 (基本 UI 見切れバグが型エラー 0 + Tier 2 LGTM + Apple Archive 成功で「通過」と誤認され TestFlight 素通り) が典型 (Reference: `dev-flow-improvement-mtg-20260520.md` §2)。

### 各課はキャラクター gate を 1 つ持つ

機械テストでは構造的に拾えない、 人間が知覚して初めて分かる合格基準 = 「キャラクター gate」 を各課が 1 つ名指しする。 自課の gate は `.github/pull_request_template.md` の「成果物 verify」 section に明記する。

| 課 | キャラクター gate の例 |
|----|----------------------|
| native / ゲーム | 実機で実際に遊べるか (画面が見える・ボタンが押せる) |
| dsp | 試聴 — 音のキャラクター |
| write / SNS | 外部読者として読めるか |
| blender | 別環境・素の状態で正しく見えるか |
| LP | 実 URL で表示・操作できるか |
| template | distribute dry-run 出力で全配布対象課に意図通りの差分か |

### 工程名で分離する

「テスト通過」 という 1 語が機械チェックと人間チェックの両方を指す曖昧さをなくす。 PR のチェックボックスを工程名で分ける:

- 「ビルド成功 ✓」 = 機械チェック工程
- 「実機で遊べた ✓」 = キャラクター gate 工程

両者は別のチェック項目として扱い、 同じ「通過」で混同しない。

### 証拠なしには「通過」と書けない

合格マークにはスクショ or 録画 (または dry-run 出力等の実行証跡) の添付を必須とする。 「見た」 と言うだけ・証拠が残っていなければ未検証扱い。 evidence なしの「通過」記載は無効。

### ツール化できない一線

最終的なキャラクター判断 (試聴・実機体験・読者視点) は構造的に人間のもの。 ツールは判断周りの段取り (スクショ撮影・差分抽出・自動操作) を減らせるが、 判断自体は代替できない。 自動化は人間判断を「省く」 のでなく「届きやすくする」 ために使う。
